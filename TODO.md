## New Tasks
- [x] Support negation patterns in .coderevignore (e.g. `!important.log` to unignore)
- [x] Optional `.gitignore` integration (load existing repo exclusions; `.coderevignore` still overrides)
- [x] Support `**` globstar patterns in `.coderevignore` (e.g. `docs/**/*.md`, `**/build/`) with proper zero-or-more-directory semantics
- [x] Anchor `.coderevignore` patterns with a leading `/` (or `./`) to the repo root (gitignore semantics): `/build/` matches top-level `build/` but not nested `src/build/`; unanchored patterns still match at any depth
- [x] Anchor `.coderevignore` patterns containing an internal `/` to the repo root and stop a lone `*` from crossing directory separators (gitignore semantics): `src/*.py` matches `src/a.py` but not `src/sub/a.py`; `doc/frotz` no longer matches `a/doc/frotz`
- [x] Support `[...]` character classes in anchored/globstar `.coderevignore` patterns (ranges `[a-z]`, negation `[!0-9]`) so they match a single non-separator char like gitignore, fixing the inconsistency where only the unanchored `fnmatch` path handled brackets
- [x] Support gitignore escape semantics in `.coderevignore`: `\#` is a literal leading `#` (not a comment), `\!` is a literal leading `!` (not a negation), and unescaped trailing whitespace is stripped
- [x] Precompile `.coderevignore` patterns once and cache them (avoid re-parsing/normalizing/recompiling every pattern for every path); invalidate the cache on `add_pattern`/`disable_defaults`
- [x] Stop `*`/`?` in unanchored `.coderevignore` patterns from crossing directory separators (gitignore FNM_PATHNAME semantics): `a?.py` no longer matches `a/.py`. Matching each path segment keeps no-slash patterns matching at any depth and lets them match a directory component (`foo` matches `a/foo/b.txt`), fixing the inconsistency where only anchored/globstar patterns enforced this
- [x] Add cost-estimation pricing for current-generation models (Claude 3.7 Sonnet; Claude 4 Opus / Opus 4.1 / Sonnet 4 / Haiku 4.5; GPT-4.1 family; o3-mini/o4-mini) with dated IDs and short aliases, and route o3/o4 through tiktoken token counting
- [x] Fix `get_model_pricing` so dated/suffixed model IDs (e.g. `gpt-4o-2024-08-06`, `gpt-4o-mini-2024-07-18`, `claude-sonnet-4-20250514`) resolve to the most specific base model via longest-prefix + token-boundary matching, instead of the first family sharing a provider prefix (previously mispriced `gpt-4o-mini` as `gpt-4`, ~200x too high on input)
- [x] Warn when cost estimation falls back to `DEFAULT_PRICING` for an unknown model: add `is_known_model()`, carry a `pricing_is_estimated` flag on `CostEstimate`, and surface it (rich warning line + JSON field) so users don't mistake a default-rate guess for an accurate figure
- [x] Resolve Anthropic hyphenated base aliases and moving `-latest` suffixes in cost estimation (e.g. `claude-3-5-sonnet-latest`, `claude-3-7-sonnet-latest`, `claude-3-5-haiku-latest`) to correct per-tier pricing instead of `DEFAULT_PRICING`; add hyphenated aliases (`claude-3-5-sonnet`, `claude-3-7-sonnet`, `claude-opus-4-1`, `claude-haiku-4-5`) and strip a trailing `-latest` to the underlying base
- [x] Add a `--max-cost` budget gate to the `estimate` command (and `CostEstimate.exceeds_budget`): exit code 2 when the estimated total cost exceeds the given USD budget so reviews can be gated in CI; budget is inclusive (at-limit passes), negative budgets are rejected, and `over_budget`/`max_cost_usd` are surfaced in the JSON output
- [x] Add a `--batch` flag / `CostEstimator(batch_mode=True)` that estimates at the 50% Batch API rate (Anthropic Message Batches / OpenAI Batch API): halves both input and output per-token prices, carries a `batch_mode` flag on `CostEstimate`, and surfaces it in the rich table and JSON output so async reviews can be budgeted correctly
- [x] Stop `*`/`?` in unanchored *directory* `.coderevignore` patterns (ending in `/`) from crossing directory separators: match each path segment individually so `foo*bar/` matches `a/fooXbar/y` but not `foo/x/bar/z`, and `a?c/` matches `abc/` but not `a/c/` — fixing the inconsistency where the directory branch still used `fnmatch` fallbacks (`*/dir/*`) that let wildcards span `/`, unlike the already-fixed non-directory glob path
- [x] Treat `**` in `.coderevignore` patterns as a globstar only when bounded by `/` or the pattern boundary (gitignore: "other consecutive asterisks are considered regular asterisks"): `a**b` now compiles to `a[^/]*b` and stays in-segment (matches `axxb`, not `ax/yb`), while `**/`, `/**`, and middle `/**/` keep crossing directories; also require a preceding boundary for the `**/`→zero-or-more-dirs rule so `a**/b` no longer spans separators
- [x] Make anchored/globstar `.coderevignore` patterns without a trailing `/` ignore a matched directory's *contents* too (gitignore: a no-slash pattern matches files and directories, and a matched directory ignores everything under it). `/build` now matches `build/foo.py`, and `docs/**/build` matches `docs/x/build/foo.py`, matching real `git check-ignore` — previously these compiled to `^pat$` and matched only the directory entry itself
- [x] Add a per-file cost breakdown to the `estimate` command (`--per-file` flag / `CostEstimator.estimate_files(detailed=True)`): attach each reviewed file's own `CostEstimate` (with its `path`) to the aggregate's `file_breakdown`, sorted most-expensive-first, so users can see which files dominate the estimate; surfaced in both the rich table and JSON (`per_file`), skipped binaries excluded, and per-file totals reconcile with the aggregate
- [x] Honor gitignore's parent-directory rule in `.coderevignore`: "it is not possible to re-include a file if a parent directory of that file is excluded." `should_ignore` now walks each ancestor directory prefix top-down and, once a directory prefix is excluded, keeps the whole subtree ignored regardless of a later `!` negation. So `build/` + `!build/keep.txt` keeps `build/keep.txt` ignored (matching `git check-ignore`), while `build/*` + `!build/keep.txt` still re-includes it because only the contents — not the directory — are excluded
- [x] Support gitignore backslash escapes for glob metacharacters in `.coderevignore`: `\*`, `\?`, `\[` now match a literal `*`/`?`/`[` instead of acting as wildcards, so `report\*.csv` matches a file literally named `report*.csv` (verified against `git check-ignore`). Previously every backslash was folded to `/` for Windows compatibility, which silently turned `report\*.csv` into the anchored pattern `report/*.csv`. A backslash before a non-metacharacter is still a Windows path separator, so `src\generated\` keeps working; use `/` when you need a wildcard after a directory. Unanchored patterns now match segments with the same escape-aware regex as the anchored/globstar path instead of `fnmatch` (which has no escape syntax), so all branches agree
- [x] Honor gitignore's directory-only (trailing-slash) rule in `.coderevignore`: "if there is a separator at the end of the pattern then the pattern will only match directories." `should_ignore(path, is_dir=None)` now takes an optional `is_dir` hint (inferred from disk when the path exists) so a dir-only pattern like `build/` matches a directory `build` but NOT a regular file named `build`, while still ignoring the directory's contents; ancestor prefixes are always treated as directories, and an unknown leaf type stays backwards-compatible
- [x] Make cache writes atomic so concurrent reviews cannot corrupt or destroy entries: `ReviewCache.set` staged JSON straight into the destination with `json.dump`, leaving a window where a concurrent reader (async batch, pre-commit hook next to a CI review, two coderev processes sharing `~/.cache/coderev`) saw a truncated file — and since `get()` treats unparseable entries as corrupt, it *deleted* the valid entry. Writes now go to a temp file in the same directory and swap in via `os.replace()` (atomic on POSIX and Windows), so a reader only ever sees the old file or the new one. A failed write (disk full, unserializable result) now leaves the previous entry intact and no temp debris. `get()` also no longer unlinks on `OSError`: a transient Windows lock during the swap is retried, then treated as a miss rather than as corruption (the old code's cleanup `unlink` could itself raise `PermissionError` and crash the reader), and `clear()` sweeps temp files left by a killed writer
- [x] Resolve provider-routed model IDs in cost estimation: routers/proxies (LiteLLM, OpenRouter, the Anthropic SDK's Bedrock/Vertex paths) prefix the model with one or more `provider/` segments (e.g. `openai/gpt-4o-mini`, `anthropic/claude-3-5-sonnet-20241022`, `openrouter/anthropic/claude-3.5-sonnet`). `_resolve_pricing` now strips the leading segment and re-resolves the remainder (recursing to peel multi-level prefixes), so these price the same as the bare model and report as `is_known_model`, instead of falling back to `DEFAULT_PRICING` (~66x off for gpt-4o-mini input). Composes with the existing `-latest` and longest-prefix resolution; an unknown model behind a prefix still falls back correctly
- [x] Add Google Gemini pricing to cost estimation (Gemini 1.5 Pro/Flash/Flash-8B, 2.0 Flash/Flash-Lite, 2.5 Pro/Flash/Flash-Lite) so models routed via LiteLLM/OpenRouter (`gemini/...`, `google/...`, `openrouter/google/...`) and dated/version suffixes (`-002`, `-001`, preview dates) resolve to real per-1M rates instead of `DEFAULT_PRICING` (~100x off for Flash); composes with existing provider-prefix stripping and longest-prefix resolution, with more specific `-lite`/`-8b` variants winning over their parent
- [x] Apply the same "a failed read is not evidence" rule to the directory-wide cache scans, which the atomic-write fix left behind. `prune_expired()` caught `OSError` and *deleted* the entry, so a transient Windows lock — exactly what a concurrent writer's `os.replace()` holds for an instant — made pruning destroy fresh, unexpired reviews; it now deletes only on evidence read off disk (parsed and past TTL, or parsed as garbage) and leaves an unreadable entry strictly alone. `stats()` called `stat()` outside its `try`, so an entry vanishing mid-scan (concurrent `clear()`/prune, second coderev process) raised `FileNotFoundError` straight out of a review; it now skips vanished files and reports locked ones under a new `unreadable_entries` field instead of miscounting them as expired. All three readers now share one `_load_entry()` loader that distinguishes CORRUPT (bytes seen, safe to drop) from UNREADABLE (bytes never seen, hands off), and both scans skip temp debris
- [x] Carry the cache's atomic-write and "a failed read is not evidence" rules over to `ReviewHistory`, where the same defect is more destructive: `add()` is a read-modify-write over a whole *month* file (load every review that month, append one, write back), so a bad read costs the month rather than one entry. `_save_month_file` truncated the destination in place, and `_load_month_file` returned `[]` for both a torn read and a transient `OSError` — so a concurrent reader (async batch, pre-commit hook beside a CI review, a second coderev process) saw a partial file, parsed nothing, and wrote back a month containing only its own entry. Verified against the pre-fix code: a lock held during `add()` destroyed 4 of 5 recorded reviews, and a reader could observe a 0-byte month file mid-write. Writes now stage to a temp file and swap in via `os.replace()` (atomic on POSIX and Windows), so a reader only ever sees the old file or the new one. Reads split by *why* they failed: read-only queries (`get_all`/`get_recent`/`get_stats`) still degrade to `[]` since they cannot destroy anything, while the read-modify-write path uses a strict loader that raises `HistoryWriteError` rather than overwrite a month it could not read. A month whose bytes *were* read and are garbage is quarantined to a `.corrupt` sidecar instead of being silently overwritten, so the content survives for manual recovery and the new entry is still recorded. Directory scans skip temp/quarantine debris, and `clear()` sweeps temp files left by a killed writer without counting them
- [x] Wire `.coderevignore` into the CLI's `collect_files` so `review`/`estimate`/`batch` honor it when expanding directories. The full ignore engine (`coderev.ignore`, with all its gitignore-semantics work — anchoring, globstar, negation, char classes, dir-only, parent-reinclude) existed and was heavily tested, but `collect_files` never consulted it: only `--exclude` fnmatch patterns filtered directory expansion. So `coderev review .` sent files the user had explicitly ignored (secrets, build output, vendored code) to the API, and `coderev estimate .` counted them — over-estimating cost. `collect_files` now loads `CodeRevIgnore.load()` once and drops ignored files (matched against a cwd-relative path so root-anchored `/build/` patterns line up with the repo root), passing `is_dir=False`. An explicitly named file on the command line is still always reviewed (an explicit path beats ignore rules, mirroring how it beats `--exclude`). Added a `--no-ignore` opt-out flag to `review` and `estimate`. Added `tests/test_cli_ignore.py` (7 cases: file drop, dir-contents drop, root-anchoring, negation re-include, opt-out, explicit-file override, compose-with-exclude)
- [x] Resolve AWS Bedrock and Google Vertex model IDs in cost estimation. The provider-routing task's docstring claimed it covered "the Anthropic SDK's Bedrock/Vertex paths," but `_resolve_pricing` only peeled slash-separated `provider/` prefixes — and Bedrock/Vertex use *dot*-separated ids. So `anthropic.claude-3-5-sonnet-20240620-v1:0`, its region-prefixed cross-region inference profiles (`us.anthropic.claude-...`, `eu.`, `apac.`), and Vertex's `claude-3-5-sonnet@20240620` all fell back to `DEFAULT_PRICING` (~3-5x off for Sonnet input, ~2x for output) and reported `is_known_model=False`. Fix: (a) peel one leading provider/region token matched *exactly* against a known set (`anthropic`, `openai`, `google`, ... | `us`, `eu`, `apac`, `ap`, `global`, `us-gov`) and recurse — so `<region>.<vendor>.` two-part prefixes strip and alias dots like `claude-3.5-sonnet` (head `claude-3`, not a prefix) are left untouched; (b) strip a trailing `:<digits>` Bedrock inference-profile version and re-resolve; (c) add `@` and `:` to the longest-prefix token-boundary set so Vertex's `@<date>` suffix resolves. Composes with existing `-latest`/slash-prefix/longest-prefix resolution; an unpriced cloud model (`amazon.nova-pro-v1:0`) still falls back correctly. Added `tests/test_cost_bedrock_vertex.py` (26 cases)
